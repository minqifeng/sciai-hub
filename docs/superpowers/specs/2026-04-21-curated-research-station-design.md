# SciAI Hub 精选研究站重设计

**日期**: 2026-04-21  
**版本**: v1.0  
**状态**: 待用户审阅  
**默认主用户**: 研究生 / 博士  
**取代文档**: `docs/superpowers/specs/2026-04-21-research-workbench-redesign-design.md`

---

## 1. 结论摘要

SciAI Hub 不再继续做“AI 科研工具目录站”，而是重构为：

`面向研究生 / 博士的 AI 科研精选站`

核心承诺从“收录很多工具”改为：

`少，但值得用；每个都告诉你为什么选、适合谁、怎么用。`

这意味着整站的默认信息架构、数据模型、页面组织和更新机制都要围绕“精选结论”而不是“全量罗列”展开。站点保留少量专题模块与编辑观察内容，但不再提供“全部工具”目录、全量分类页、排序筛选页或大而全的工具库体验。

---

## 2. 背景与问题

当前站点虽然已经往 research workbench 方向收束，但仍残留明显的目录站心智：

- 首页依然保留完整工具网格和大量分类筛选能力。
- 用户仍会被引导去“找所有工具”，而不是先获得高质量判断。
- 许多条目对研究主用户并不重要，却与高价值工具出现在同一层级。
- 单个工具虽然有描述，但缺乏“为什么推荐 / 怎么使用 / 什么情况下不要用”的编辑说明。
- 自动更新目前更接近“拉取数据”，还不是“候选变化 -> 编辑复核 -> 发布结论”的精选站工作流。

对研究生 / 博士来说，这类问题会直接伤害站点价值：他们需要的是决策支持，而不是一个需要再次筛选的大仓库。

---

## 3. 产品定位

### 3.1 一句话定位

`给研究生 / 博士的 AI 科研精选站`

### 3.2 核心承诺

站点必须同时兑现以下三件事：

1. 按研究任务给出高质量精选，而不是按工具品类堆列表。
2. 对每个入选工具给出明确使用说明和适用边界。
3. 对每条精选结论给出可见的复核状态和更新时间。

### 3.3 非目标

以下内容明确不做：

1. 不做“全量工具大全”。
2. 不做面向所有职业人群的泛 AI 导航站。
3. 不做滚动信息流或热点站。
4. 不做依赖自动抓取直接上首页的无人值守推荐系统。

## 4. 设计原则

### 4.1 编辑优先

所有首页和核心页面默认展示的内容都必须是“编辑结论”。即使数据来自自动同步，也只能先进入候选层，经过复核后再进入公开展示。

### 4.2 任务优先

研究主用户的目标不是“浏览工具”，而是“完成文献综述、实验分析、论文写作、投稿回复”等任务。页面应优先组织任务与剧本，而不是工具分类。

### 4.3 说明优先

每个精选工具都必须能回答：

- 为什么推荐
- 最适合谁
- 不适合谁
- 3 步怎么用
- 用它替代什么

如果写不出这些，就不应进入精选区。

### 4.4 少量高信号

整站宁可内容更少，也不能为了显得丰富而保留低质量条目。用户记住的应该是“这里替我做了判断”，而不是“这里东西很多”。

### 4.5 可信状态可见

每个重要对象都应能显示：

- 来源
- 最近复核时间
- 当前状态
- 是否近期发生变化

## 5. 信息架构

整站主导航重构为：

- 首页
- 本周精选
- 研究剧本
- 工具评测
- 更新记录

如需保留辅助内容，只能进入次级导航：

- 编辑观察
- 方法工具
- 站点说明 / 提交建议

### 5.1 首页

首页只展示最值得进入的精选内容，模块顺序固定为：

1. Hero
2. 本周精选
3. 研究剧本
4. 高频必备
5. 方法工具
6. 更新记录

首页不再出现：

- 全量工具网格
- 工具分类 tab
- 价格筛选
- 国产 / 海外筛选
- 排序器
- “查看全部工具”按钮

### 5.2 本周精选

本周精选页是“编辑封面”，不是归档页。它只放本周最值得优先查看的 3-8 个对象，可以是：

- 单个工具
- 工具组合
- 研究剧本
- 专题观察

每个对象都必须给出明确推荐理由和近期变化说明。

### 5.3 研究剧本

研究剧本页按高频研究任务组织，例如：

- 文献综述
- 实验分析
- 论文写作
- 投稿回复
- 组会汇报
- 开题准备

每个剧本只配 2-4 个工具，并说明推荐顺序、替代关系和完成路径。

### 5.4 工具评测

工具评测页不做全量索引，而是做深度评测集合。每篇评测必须回答：

- 推荐结论
- 适用场景
- 使用门槛
- 不适用场景
- 替代方案

### 5.5 更新记录

更新记录页承担“准确消息”的可见化职责，记录：

- 新增精选
- 下架精选
- 结论修订
- 复核完成
- 数据状态变化

## 6. 首页设计

### 6.1 Hero

Hero 的职责是一次性说清站点价值，而不是堆卖点。

左侧：

- 一句话定位
- 一段副标题
- 一个任务输入框
- 3-5 个主任务入口

右侧：

- 本周主张
- 最近复核时间
- 当前已复核内容数量
- 最近一次重大变化

任务输入框不再返回全量工具列表，而是返回“精选建议”或“研究剧本”。

### 6.2 本周精选区

首页本周精选默认展示 3 个主卡 + 3-6 个副卡。每张卡片只呈现结论型信息：

- 工具 / 主题名称
- 一句话定位
- 为什么值得选
- 最适用任务
- 复核时间

### 6.3 研究剧本区

每个剧本卡片包含：

- 任务名
- 适用阶段
- 建议工具数量
- 一句话流程

点击后进入剧本详情，而不是展开更多工具列表。

### 6.4 高频必备区

这是常青精选区，用于放置研究主用户的稳定高频工具。数量控制在 6-9 个。这里的推荐必须强调“长期可用性”和“上手成本”。

### 6.5 方法工具区

保留并升级原有专题模块：

- 选刊助手
- 引文核查
- PaperDeck
- 统计方法

它们不再与普通工具混排，而作为“方法工具”独立展示。

### 6.6 更新记录区

更新记录在首页展示最近 3-5 条，格式统一为：

- 变化类型
- 对象名称
- 变化摘要
- 时间

## 7. 精选对象的数据模型

站点不再使用“全量收录工具库”作为核心对象，而使用 `CuratedEntry`。

建议字段如下：

```text
id
slug
type                // tool | workflow | module | note
name
positioning         // 一句话定位
bestFor             // 最适合谁
avoidIf             // 不适合谁
whySelected         // 为什么入选
howToUse            // 3 步使用说明
scenes              // 适用场景数组
workflowRole        // 在剧本中的角色
alternatives        // 替代方案
priceNote
difficulty          // easy | medium | hard
confidence          // high | medium | low
reviewStatus        // reviewed | pending_review | changed_recently
reviewedAt
changedAt
sourceLabel
sourceUrl
tags
url
```

### 7.1 入选标准

一个工具或对象只有满足至少 3 项时才允许进入精选池：

1. 对研究生 / 博士真实高频
2. 能写出明确推荐场景
3. 有稳定入口或稳定可达性
4. 有明显优于替代品的理由
5. 能写出具体使用说明
6. 对中文科研用户足够友好或价值足够高

### 7.2 淘汰标准

以下对象应从首页或精选站移除：

- 长期不再更新且无稳定价值
- 说明写不清楚
- 与已有精选高度重复
- 对主用户帮助有限
- 只看上去新，但不具备长期意义

## 8. 卡片与详情结构

### 8.1 首页精选卡

首页卡片固定展示：

- 名称
- 一句话定位
- 为什么值得选
- 推荐场景
- 复核状态

### 8.2 详情页 / 详情弹层

单个精选对象的详情必须固定为以下结构：

1. 编辑结论
2. 最适合谁
3. 不适合谁
4. 3 步使用说明
5. 推荐工作流组合
6. 替代方案
7. 价格 / 门槛
8. 最近复核时间
9. 最近一次变更说明

这样“使用说明”成为主能力，而不是附属字段。

---

## 9. 搜索与导航行为

### 9.1 搜索行为

搜索对象只包括：

- 精选工具
- 研究剧本
- 方法工具
- 评测内容

搜索结果应按“结论匹配度”而不是“名称匹配数量”排序。

### 9.2 不再保留的导航行为

以下交互需要下线：

- 浏览全部工具
- 按价格浏览全库
- 按国家 / 地区浏览全库
- 通过排序器查看全量条目

### 9.3 仍然保留的导航行为

- 从首页进入本周精选
- 从任务进入研究剧本
- 从精选卡进入详情页
- 从详情页进入外部工具官网

---

## 10. 自动更新与复核流程

站点更新机制必须从“自动拉数据”升级为“候选变化 -> 编辑复核 -> 公开发布”。

### 10.1 日更自动层

职责：

- 检查工具链接是否可达
- 检查 manifest 新鲜度
- 收集 news / models / github 的候选变化
- 标记可能需要复核的精选对象

### 10.2 周更编辑层

职责：

- 重新评估本周精选
- 更新研究剧本中的工具组合
- 修订工具使用说明
- 生成更新记录

### 10.3 发布原则

自动同步结果不直接进入首页。只有被标记为 `reviewed` 的对象才能进入：

- 首页
- 本周精选
- 研究剧本
- 工具评测

### 10.4 状态表达

建议统一状态：

- `reviewed`
- `pending_review`
- `changed_recently`
- `degraded`

状态必须在 UI 中直接可见。

## 11. 视觉方向

### 11.1 总体方向

视觉方向统一为：

`编辑型科研精选站`

不是工具商城，也不是信息流，而是像一份持续维护的研究编辑产品。

### 11.2 气质要求

- 克制
- 干净
- 有编辑判断
- 信息密度高但不拥挤
- 让“结论”成为视觉中心

### 11.3 视觉重点

1. 弱化数量感，强化判断感。
2. 首页卡片更少、更厚、更有解释。
3. 状态信息以细条或信息行展示，而非大面积装饰。
4. 颜色系统优先表达可信度与状态，而不是追求花哨。

### 11.4 版式原则

- 首页尽量避免长网格。
- 重要区块优先双栏或 editorial layout。
- 每个模块只承担一种信息任务。
- 不出现“很多类似卡片平铺到底”的目录站视觉。

---

## 12. 现有模块的去留

### 12.1 删除或停用

- 全量工具主网格作为站点核心入口
- 全量分类 tab
- 价格筛选
- 国产 / 海外筛选
- 默认排序器
- “查看全部工具”导向

### 12.2 保留但改造

- `recommendSection`
  - 从“推荐几个工具”改为“返回精选方案与研究剧本”。
- `news / models / github`
  - 降级为“编辑观察”或“更新记录的一部分”。
  - 不再是独立的大频道入口。

### 12.3 保留并升级

- `journal`
- `cite-check`
- `paperdeck`
- `stats`

这些模块从“普通分类内容”升级为“方法工具”。

## 13. 错误处理与降级策略

### 13.1 数据缺失

如果候选数据无法同步：

- 首页只展示已复核的稳定内容
- 更新记录显示“候选同步异常”
- 不允许用占位内容伪装成最新结论

### 13.2 单个精选对象缺字段

如果缺少以下关键字段之一，则不得进入首页：

- whySelected
- howToUse
- reviewedAt

可以保留在内部候选区，但不能公开展示。

### 13.3 外部链接失效

如果链接失效：

- 标记为 `degraded`
- 自动进入待复核状态
- 从首页和本周精选移除

---

## 14. 验收标准

改版完成后，应满足以下验收条件：

1. 首页不再出现“全部工具目录”体验。
2. 站内不存在“查看全部工具”的主路径。
3. 默认公开展示的都是精选对象，而不是全量工具条目。
4. 每个精选对象都具有明确的使用说明。
5. 用户能够先按任务进入，再看到推荐工具。
6. 每个首页重点对象都能看到复核或更新时间。
7. 原有方法工具模块仍可达，但不再破坏精选站主叙事。

---

## 15. 测试策略

### 15.1 内容测试

- 检查首页是否仍存在目录站文案
- 检查精选对象是否都具备说明字段
- 检查剧本页是否只使用少量高质量工具

### 15.2 交互测试

- 搜索仅返回精选对象
- 首页卡片跳转正确
- 研究剧本与方法工具入口正常

### 15.3 状态测试

- reviewed / pending_review / degraded 状态显示正确
- reviewedAt 缺失时不得公开展示

### 15.4 回归测试

- 原有 `journal / cite-check / paperdeck / stats` 仍可进入
- 不再存在旧的工具筛选 / 排序主路径

---

## 16. 后续实施边界

本 spec 只定义以下内容：

- 精选站的产品定位
- 信息架构
- 数据模型
- 首页与详情页结构
- 自动更新与复核策略
- 现有模块去留原则

本 spec 不直接规定：

- 具体代码分工
- 分阶段实施顺序
- 每一步的提交粒度

这些内容应由后续 implementation plan 单独定义。
