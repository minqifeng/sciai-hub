# SciAI Hub 研究工作台化重构设计

**日期**: 2026-04-21
**版本**: v1.0
**状态**: 待用户审阅
**默认主用户**: 研究生 / 博士

---

## 背景与目标

SciAI Hub 当前已经具备较大的内容体量和多个科研能力模块，但首页同时承担工具导航、工作流入口、消息聚合、模型榜单和趋势展示等多重职责，导致主线不清、可信度表达不足、自动更新机制不完整。

本次重构不再把产品定义为“更大的 AI 工具目录”，而是明确收束为：

`面向研究生/博士的科研 AI 工作台入口`

核心目标：

1. 让用户按科研任务快速找到合适工具。
2. 让用户清楚知道推荐、消息和榜单的来源、更新时间与验证状态。
3. 让自动同步与降级机制成为正式产品能力，而不是前端临时拉取行为。
4. 让整站视觉从“工具集市”提升为“可信、克制、长期可用的研究控制台”。

---

## 产品定位

### 主承诺

首页主承诺统一为：

`按科研任务，快速找到合适的 AI 工具。`

副标题负责解释而不是堆叠卖点：

`覆盖文献检索、阅读、分析、绘图、写作与投稿，并展示来源、更新时间与验证状态。`

### 用户要解决的问题

1. 当前科研任务下一步该做什么。
2. 这一类任务最适合用哪些工具。
3. 这些推荐和消息当前是否可靠。

### 非目标

1. 不追求做成实时资讯流产品。
2. 不追求覆盖所有 AI 领域与所有人群。
3. 不做一次性全模块重写。

## 首页信息架构

### 首页主线

首页只讲一条主线：

`按科研任务找工具，并看到可信度。`

### 首页模块顺序

1. `Hero 首屏`
2. `科研工作流导航`
3. `精选工具区`
4. `完整工具库`
5. `次级内容入口`

### Hero 首屏

左栏放：

- 主标题
- 搜索框
- 5 个主任务入口
- 简短行动提示

主任务入口固定为：

- 找文献
- 读论文
- 做分析
- 做图
- 写论文

右栏放可信度摘要，不再放营销式规模数字。建议展示：

- 数据覆盖
- 最近同步时间
- 已验证来源数
- 当前缺口或待补充项

### 科研工作流导航

压缩为单条科研路径：

`问题定义 -> 文献 -> 阅读 -> 分析 -> 可视化 -> 写作/投稿`

不再保留大量同层级重复入口。

### 精选工具区

精选区优先于完整工具库。每张卡片必须解释：

- 为什么推荐
- 适合哪个任务阶段
- 来源
- 更新时间
- 验证状态

### 完整工具库

完整工具库保留，但降到首页下半段，不再担任首页主角。

### 次级内容入口

以下内容保留，但从首页主舞台降级：

- 行业消息
- 模型榜单
- GitHub 趋势
- Use Cases
- 统计方法扩展入口

## 数据可信度层

### 原则

前端不再承担“真假实时”的责任。产品采用：

`定时同步 -> 校验 -> 生成可发布数据 -> 前端展示`

### 通用可信度字段

所有重要实体尽量具备以下字段：

- `id`
- `entity_type`
- `title` / `name`
- `summary`
- `source`
- `source_url`
- `record_type`
- `captured_at`
- `last_verified_at`
- `freshness_state`
- `confidence`
- `status`

### 关键字段约定

- `record_type`: `editorial` / `snapshot` / `live_sync` / `fallback`
- `freshness_state`: `fresh` / `stale` / `degraded`
- `confidence`: 低 / 中 / 高

### 分数据类型治理策略

#### 工具库

- 以人工整理为主
- 定时做链接检测、字段补全、重复项检查
- 追求稳定和可解释，不追求分钟级更新

#### 行业消息

- 仅展示真实抓取或人工整理结果
- 人工整理内容必须明确标记为 `editorial`
- 禁止模板化内容伪装成实时消息

#### 模型榜单

拆为两层：

- `weekly snapshot`
- `catalog sync`

榜单快照与目录同步必须分开表达。

#### GitHub 趋势

作为外部信号展示，必须显式说明：

- 数据窗口
- 抓取时间
- 筛选条件

## 自动同步与验证架构

### 目标

用仓库中的定时任务生成正式数据文件，不再通过同步脚本改写前端代码常量。

### 推荐任务

1. `news-sync`
   - 北京时间 08:00 / 14:00 / 20:00
   - 同步 RSS 与可信新闻源
   - 负责去重、标准化、产出新闻数据

2. `models-catalog-sync`
   - 北京时间 10:00 每天
   - 同步公开模型目录
   - 生成模型目录数据与 freshness manifest

3. `models-snapshot-freeze`
   - 每周一北京时间 10:00
   - 冻结周榜快照

4. `github-trending-sync`
   - 北京时间 09:00 / 21:00
   - 同步 GitHub 趋势项目与抓取窗口信息

5. `validation`
   - PR 时运行
   - 夜间定时运行
   - 负责 schema、重复 ID、死链、字段缺失、freshness 过期检查

### 降级策略

- 同步成功：显示最新同步结果
- 同步失败但存在旧数据：显示旧数据并标记 `degraded`
- 无数据：展示空态与原因

禁止伪造实时结果填补空白。

## UI 视觉系统

### 总体方向

视觉方向统一为：

`Premium Research Console`

目标气质：

- 克制
- 清晰
- 有秩序
- 编辑感
- 不依赖炫技

### 排版与字体

建议分层：

- 中文正文：`Noto Sans SC` 或 `Source Han Sans SC`
- 英文与技术标签：`IBM Plex Sans`
- 时间戳与状态信息：`IBM Plex Mono`

原则：

- 标题更短、更硬
- 副标题负责解释，不堆关键词
- 卡片描述控制在 1 到 2 行
- 用结构化信息替代长段文案

### 颜色系统

- 主色：深蓝，用于主要操作与主要导航
- 辅色：克制绿色，仅用于可信状态
- 中性色：承担页面结构
- 警示色：琥珀，用于过期 / 降级 / 待验证
- 错误色：红色，仅用于失败

颜色优先表达状态，不承担装饰职责。

### 首页视觉改造重点

1. 首屏改为强双栏结构
2. 右栏升级为可信度状态板
3. 工作流条与 taxonomy 合并为单条科研路线
4. 精选工具区强化编辑感
5. 完整工具库降低视觉噪声

### 通用卡片规则

- 一张卡片只表达一个主信息
- 元数据只保留最关键项
- 标签最多 3 个
- 每张重要卡片必须具备状态行

统一新增 `trust row`，用于展示：

- 来源
- 更新时间
- 验证状态
- 置信度

### 动效原则

保留：

- 轻微入场
- 卡片 hover 抬升
- 筛选切换过渡
- 局部状态反馈

避免：

- 大面积漂浮
- 重度视差
- 多层发光
- 每个区块都在动

### 移动端原则

移动端首页只保留：

- 搜索
- 5 个任务入口
- 可信度摘要
- 精选工具
- 进入完整工具库按钮

新闻、模型榜单、GitHub 趋势收纳到二级入口。

## 重构边界

### 保留的能力模块

以下模块先保留能力本体，不做优先级最高的重写：

- 研究图谱
- 论文检索
- 选刊助手
- 引文核查
- PaperDeck
- 统计方法库

这些模块对应真实科研动作，应保留为二级能力页。

### 优先重做部分

1. 首页骨架
2. 一级导航与侧边栏
3. 工具卡片体系
4. 新闻与趋势内容层
5. 数据读取方式
6. 自动同步与验证流程

### 推荐代码结构目标

#### App Shell

负责：

- 导航
- 页面骨架
- 全局状态
- 主题

#### Feature Modules

每个二级能力页独立负责自身 UI 与行为。

#### Data Layer

只负责：

- 读取数据文件
- 读取 manifest
- 处理 freshness / provenance / fallback

#### Sync & Validation Layer

只负责：

- 同步
- 标准化
- 去重
- 校验
- 生成可发布数据

## 实施阶段

### Phase 0：稳定性修复

先修当前明确存在的正确性问题：

- Hero 快捷入口报错
- 重复初始化
- 年份筛选 bug
- 同步脚本只提交部分文件的问题

### Phase 1：数据与可信度底座

建立：

- `data/` 目录
- manifest
- provenance / freshness 字段
- validation

### Phase 2：首页与导航重做

重做：

- 首屏
- 工作流导航
- 一级导航
- 侧边栏层级
- 精选工具区

### Phase 3：卡片与信任系统

统一重做：

- 工具卡
- 新闻卡
- 模型卡
- GitHub 卡
- trust row
- freshness / degraded / snapshot 表达

### Phase 4：二级能力页接入新壳

把现有能力模块接入新的导航、状态层和视觉系统。

### Phase 5：定时同步上线

完成：

- news-sync
- models-catalog-sync
- models-snapshot-freeze
- github-trending-sync
- validation

---

## 禁止事项

重构过程中明确不做：

1. 一次性全模块重写
2. 先做纯视觉翻新
3. 继续扩充首页信息密度
4. 把客户端临时拉取当成可靠实时机制
5. 用模板内容冒充实时内容

---

## 验收标准

完成后至少满足：

1. 首页首次打开即可清楚表达主任务。
2. 用户可以明确区分实时、快照、编辑整理和降级展示。
3. 新闻、模型、趋势内容有明确来源与更新时间。
4. 自动同步失败时页面能诚实降级。
5. 首页视觉更高级，但不会掩盖数据问题。
6. 二级能力页保留并更容易进入。
